# Toast 商家后台目录结构 — 信息架构分析

> 基于 `Toast结构分析.md` 与 `Toast结构分析-中文.md` 对照归纳，从信息架构（IA）视角说明分类思路、维度、理论参照、可借鉴点与改进空间。  
> 说明：Toast 未公开其内部 IA 方法论，文中「理论基础」为学界/业界常用框架，用于解释其结构为何常见、可对标。

---

## 1. 整体分类思路（四个「分组」）

从脑图中的 **分组一～四** 可归纳出一套典型的 **「前台驾驶舱 → 业务域配置 → 平台与账户 → 运维/内部」** 分层：

| 分组 | 内容重心 | 角色隐喻 |
|------|----------|----------|
| **分组一** | 首页指标、快捷入口、**报表大树**、探索产品 | 店长/运营：**先看数、再下钻报表** |
| **分组二** | 员工、菜单、外带外送、支付、顾客、营销、前厅、后厨、订位等 **业务模块** | 各职能：**按业务域做日常配置与运营** |
| **分组三** | 集成、商店、**Toast 账户**（账单、企业/门店、合规）、支持中心 | 老板/总部/IT：**订阅、组织、合规、求助** |
| **分组四** | 内部工具、设备中心 | 运维/实施：**偏技术侧与设备** |

**一句话：** 先把 **「每天盯盘 + 看报表」** 放在最前；中间大块是 **按餐饮业务域切分的能力**；再往外是 **租户级、与「做菜卖饭」弱相关但必有的平台能力**；最外是 **设备与内部工具**。

---

## 2. 归类维度（多维度混合，非单一标准）

Toast 的目录 **不是**「只按岗位」或「只按数据域」，而是 **多维度叠加**，主要包括：

1. **业务域 / 价值流（主维度）**  
   销售、人力、菜单、支付、现金与损耗、账务、后厨、营销、订位等，与餐饮 **从点餐到结账、从排班到报表** 的自然分工一致，接近 **领域驱动中「按子域切模块」** 的粗粒度做法。

2. **任务类型：读 vs 配**  
   - **读**：首页卡片、各类 Reports、模块下的 Related reports。  
   - **配**：各模块下的设置、规则、集成项。  
   同一对象（如工时、小费）会同时出现在 **报表侧** 与 **员工/薪资模块侧**，体现 **「分析在报表、源头在配置」** 的双入口。

3. **产品/子产品边界**  
   在线点餐、配送、Payroll、会员、广告等常以 **产品线** 挂在对应域下；导航也承担 **「卖什么、客户买了什么」** 的展示（如「探索产品」「Toast 新品」）。

4. **组织层级：门店 vs 集团**  
   报表中的「集团总览」「分门店」「多门店」等，体现 **连锁** 视角嵌入导航。

5. **顾客旅程触点（辅助维度）**  
   外带外送、订位与等位、顾客、会员、反馈等，按 **顾客侧触点** 聚类，与纯后台部门视角混合使用。

6. **稳定性与权限敏感度**  
   分组三把 **低频、高权限、跨门店** 的账户/集成/合规与 **分组二的高频运营** 分开，降低日常导航噪音。

**结论：** 目录是 **「业务域为主 + 读/配分离 + 产品线嵌入 + 多店」** 的混合模型；很难用单一标签概括，这在大型企业后台中很常见。

## 3. 可挂靠的理论基础（解释用，非 Toast 官方表述）

| 框架 | 与 Toast 结构的对应关系 |
|------|-------------------------|
| **信息架构（IA）** | 层级、广度与深度；首页 + 模块的 **中心辐射** 结构。 |
| **层次任务分析（HTA）/ 工作流** | 先「看经营结果」再「进模块改规则」的路径。 |
| **Jobs-to-be-done** | 「日结对账」「改菜单」「看人力」等对应不同分支。 |
| **服务蓝图** | 前厅/后厨/顾客触点 vs 中台报表 vs 后台账户与集成。 |
| **DDD 子域（粗粒度）** | 菜单、支付、人力等作为 **限界上下文** 的近似映射（导航不必等于 DDD）。 |
| **分析型 vs 操作型任务** | 报表（分析）与设置/规则（操作）在树中的分岔。 |

## 4. 值得借鉴之处

1. **「首页 + 报表中心」前置**  
   餐饮决策者高频需求是 **看数**，驾驶舱 + 大型报表树符合 **「先结论后行动」**。

2. **报表按业务主题组织**  
   销售 / 人力 / 菜单 / 支付 / 现金与损耗 / 账务 / 后厨 / 营销等，与经营者心智中的 **账本维度** 一致，降低学习成本。

3. **模块内「相关报表 / 相关设置」**  
   同一业务域内 **读、配、查数** 距离近，减轻「改完设置再去别处找报表」的割裂。

4. **平台级能力收口（分组三）**  
   集成、账单、企业/门店、合规、支持与 **日常点菜卖饭** 分流，一线用户不易被合同与账户类信息淹没。

5. **设备与内部工具单独一层（分组四）**  
   实施、IT、运维与店长日常 **分流**，权限与心智更清晰。

6. **商业化与增购位**  
   「探索产品」、新品与行业对标等，体现导航在 **增购与价值证明** 上的作用；可借鉴其 **位置与频次控制**（是否打扰需单独评估）。

## 5. 有待提高或需注意的张力

1. **重复入口**  
   同一指标或能力（如销售汇总、人力、工时）出现在 **首页快捷、报表、模块、Related reports** 等多处——对熟手是快捷，对新手易产生 **「以哪为准」** 的困惑；需 **主入口 / 别名** 的语义设计或统一命名。

2. **分组边界认知重叠**  
   「报表总览」与「模块内报表」、分组一与分组二在 **读数** 上可能重叠；需依赖 **面包屑、标签、说明** 补强「信息气味」，单靠树形不够。

3. **命名与层级颗粒度不一致**  
   导航项与 **营销长句** 混用（如排班下整段说明）会 **拉长树、削弱可扫描性**；全大写标签（如 REPORTS）与句式不统一也会增加噪音。

4. **跨域流程被模块切开**  
   真实流程常是 **订单 → 支付 → 对账 → 报表** 跨多域；按域切导航 **必然** 带来 **流程向** 的二次学习。常见补法：**场景化工作台、最近使用、收藏**，而非仅靠目录深度。

5. **角色维度不突出**  
   目录较少以 **岗位**（收银、店长、总部）作为一级维度；若目标产品 **多角色差异大**，需额外考虑 **角色首页或权限视图**，不宜照搬其树。

6. **地域与合规**  
   拒付、EBT、Capital 等 **强绑定特定市场**；借鉴 **骨架** 时，子节点须 **本地化**，避免直接复制文案与能力名。

---

## 6. 落地建议（给自建餐饮后台）

- **可借鉴的骨架：** 驾驶舱 + 按餐饮业务域分模块 + 报表主题与配置同源 + 平台账户外置。  
- **建议补齐：** 跨域流程引导、重复入口的语义统一、按 **角色/门店规模** 的导航变体或快捷层，避免只学到「树很深」而找不到 **主路径**。
