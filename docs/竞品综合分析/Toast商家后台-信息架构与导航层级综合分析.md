# 第一部分：信息架构与归类维度

> 从信息架构（IA）视角说明分类思路、维度、理论参照、可借鉴点与改进空间。  
> Toast 未公开其内部 IA 方法论，文中「理论基础」为学界/业界常用框架，用于解释其结构为何常见、可对标。

---

## 1.1 整体分类思路（四个「分组」）

从脑图中的 **分组一～四** 可归纳出一套典型的 **「前台驾驶舱 → 业务域配置 → 平台与账户 → 运维/内部」** 分层：

| 分组 | 内容重心 | 角色隐喻 |
|------|----------|----------|
| **分组一** | 首页指标、快捷入口、**报表大树**、探索产品 | 店长/运营：**先看数、再下钻报表** |
| **分组二** | 员工、菜单、外带外送、支付、顾客、营销、前厅、后厨、订位等 **业务模块** | 各职能：**按业务域做日常配置与运营** |
| **分组三** | 集成、商店、**Toast 账户**（账单、企业/门店、合规）、支持中心 | 老板/总部/IT：**订阅、组织、合规、求助** |
| **分组四** | 内部工具、设备中心 | 运维/实施：**偏技术侧与设备** |

**一句话：** 先把 **「每天盯盘 + 看报表」** 放在最前；中间大块是 **按餐饮业务域切分的能力**；再往外是 **租户级、与「做菜卖饭」弱相关但必有的平台能力**；最外是 **设备与内部工具**。

---

## 1.2 归类维度（多维度混合，非单一标准）

Toast 的目录 **不是**「只按岗位」或「只按数据域」，而是 **多维度叠加**，主要包括：

1. **业务域 / 价值流（主维度）**  
   销售、人力、菜单、支付、现金与损耗、账务、后厨、营销、订位等，与餐饮 **从点餐到结账、从排班到报表** 的自然分工一致，接近 **领域驱动中「按子域切模块」** 的粗粒度做法。

2. **任务类型：读 vs 配**  
   - **读**：首页卡片、各类 Reports、模块下的 Related reports。  
   - **配**：各模块下的设置、规则、集成项。  
   同一对象（如工时、小费）会同时出现在 **报表侧** 与 **员工/薪资模块侧**，体现 **「分析在报表、源头在配置」** 的双入口。

3. **产品/子产品边界**  
   在线点餐、配送、Payroll、会员、广告等常以 **产品线** 挂在对应域下；导航也承担 **「卖什么、客户买了什么」** 的展示（如「探索产品」「Toast 新品」）。

4. **组织层级：门店 vs 集团**  
   报表中的「集团总览」「分门店」「多门店」等，体现 **连锁** 视角嵌入导航。

5. **顾客旅程触点（辅助维度）**  
   外带外送、订位与等位、顾客、会员、反馈等，按 **顾客侧触点** 聚类，与纯后台部门视角混合使用。

6. **稳定性与权限敏感度**  
   分组三把 **低频、高权限、跨门店** 的账户/集成/合规与 **分组二的高频运营** 分开，降低日常导航噪音。

**结论：** 目录是 **「业务域为主 + 读/配分离 + 产品线嵌入 + 多店」** 的混合模型；很难用单一标签概括，这在大型企业后台中很常见。

---

## 1.3 可挂靠的理论基础（解释用，非 Toast 官方表述）

| 框架 | 与 Toast 结构的对应关系 |
|------|-------------------------|
| **信息架构（IA）** | 层级、广度与深度；首页 + 模块的 **中心辐射** 结构。 |
| **层次任务分析（HTA）/ 工作流** | 先「看经营结果」再「进模块改规则」的路径。 |
| **Jobs-to-be-done** | 「日结对账」「改菜单」「看人力」等对应不同分支。 |
| **服务蓝图** | 前厅/后厨/顾客触点 vs 中台报表 vs 后台账户与集成。 |
| **DDD 子域（粗粒度）** | 菜单、支付、人力等作为 **限界上下文** 的近似映射（导航不必等于 DDD）。 |
| **分析型 vs 操作型任务** | 报表（分析）与设置/规则（操作）在树中的分岔。 |

---

## 1.4 值得借鉴之处

1. **「首页 + 报表中心」前置**  
   餐饮决策者高频需求是 **看数**，驾驶舱 + 大型报表树符合 **「先结论后行动」**。

2. **报表按业务主题组织**  
   销售 / 人力 / 菜单 / 支付 / 现金与损耗 / 账务 / 后厨 / 营销等，与经营者心智中的 **账本维度** 一致，降低学习成本。

3. **模块内「相关报表 / 相关设置」**  
   同一业务域内 **读、配、查数** 距离近，减轻「改完设置再去别处找报表」的割裂。

4. **平台级能力收口（分组三）**  
   集成、账单、企业/门店、合规、支持与 **日常点菜卖饭** 分流，一线用户不易被合同与账户类信息淹没。

5. **设备与内部工具单独一层（分组四）**  
   实施、IT、运维与店长日常 **分流**，权限与心智更清晰。

6. **商业化与增购位**  
   「探索产品」、新品与行业对标等，体现导航在 **增购与价值证明** 上的作用；可借鉴其 **位置与频次控制**（是否打扰需单独评估）。

## 1.5 有待提高或需注意的张力

1. **重复入口**  
   同一指标或能力（如销售汇总、人力、工时）出现在 **首页快捷、报表、模块、Related reports** 等多处——对熟手是快捷，对新手易产生 **「以哪为准」** 的困惑；需 **主入口 / 别名** 的语义设计或统一命名。

2. **分组边界认知重叠**  
   「报表总览」与「模块内报表」、分组一与分组二在 **读数** 上可能重叠；需依赖 **面包屑、标签、说明** 补强「信息气味」，单靠树形不够。

3. **命名与层级颗粒度不一致**  
   导航项与 **营销长句** 混用（如排班下整段说明）会 **拉长树、削弱可扫描性**；全大写标签（如 REPORTS）与句式不统一也会增加噪音。

4. **跨域流程被模块切开**  
   真实流程常是 **订单 → 支付 → 对账 → 报表** 跨多域；按域切导航 **必然** 带来 **流程向** 的二次学习。常见补法：**场景化工作台、最近使用、收藏**，而非仅靠目录深度。

5. **角色维度不突出**  
   目录较少以 **岗位**（收银、店长、总部）作为一级维度；若目标产品 **多角色差异大**，需额外考虑 **角色首页或权限视图**，不宜照搬其树。

6. **地域与合规**  
   拒付、EBT、Capital 等 **强绑定特定市场**；借鉴 **骨架** 时，子节点须 **本地化**，避免直接复制文案与能力名。

---

## 1.6 落地建议（给自建餐饮后台）

- **可借鉴的骨架：** 驾驶舱 + 按餐饮业务域分模块 + 报表主题与配置同源 + 平台账户外置。  
- **建议补齐：** 跨域流程引导、重复入口的语义统一、按 **角色/门店规模** 的导航变体或快捷层，避免只学到「树很深」而找不到 **主路径**。

# 第二部分：一级 / 二级导航 — 命名、场景、链路与角色分析

---

## 2.1 分析方法与层级定义

| 术语 | 本文含义 | 典型界面映射（推断） |
|------|----------|----------------------|
| **分组（一～四）** | 分析师划分的四大 **心智与权限边界** | 可能对应侧栏 **分区标题**、或仅存在于站点地图 |
| **一级导航** | `# 分组` 下的 **第一种子节点** | 侧栏 **主入口** 或 **顶部主 Tab** |
| **二级导航** | 一级节点下的 **再一层子节点** | 进入模块后的 **左侧子菜单** 或 **页面内 Tab** |
| **更深节点** | 报表名、具体设置项 | 三级以下：多为 **落地页、报表名、表单区块** |

**归类维度回顾（与本表配合阅读）：** 业务域为主，辅以「读（报表）/ 配（设置）」、产品线、多门店、顾客触点等（详见 **本文第一部分 §1.2**）。

## 2.2 分组一：驾驶舱 + 报表 + 增长入口

### 2.2.1 分组定位

| 维度 | 说明 |
|------|------|
| **整体使用场景** | 开店后 **每日先看经营结果**、按需下钻报表；顺带了解 **可加购产品**。 |
| **主要使用角色** | **店长、运营、老板**（读数为主）；**总部数据分析**（报表总览）；实施期 **IT**（报表相关「相关-设置」中的导出与密钥）。 |
| **典型链路** | 登录 → **首页** 看 KPI → 点击卡片/快捷入口 → 具体报表或子产品 → 必要时到 **报表** 树按主题查历史/明细 → 「探索产品」了解增值能力。 |

### 2.2.2 一级导航：首页

| 项目 | 内容 |
|------|------|
| **命名定义** | **Home / 首页**：门店经营 **当日/近期** 的 **仪表盘**，聚合指标、快捷入口与增长位。 |
| **使用场景** | 开市前/打烊后 **扫一眼** 销售、人力、折扣；需要时 **一键跳进** 常用报表或配置（菜单、退款、工时等）。 |
| **二级导航（节选）与含义** | 见下表。 |
| **使用角色** | 店长、值班经理（高频）；区域经理（多店时看对比，若产品支持）。 |
| **使用链路（示例）** | ① 看「净销售额-今日」→ ② 点「销售汇总」快捷 → ③ 进入报表详情；或 ① 点「Toast 新品」→ ② 了解附加费/Checking 等增值。 |

#### 首页下二级导航简表

| 二级（及典型子项） | 命名意图 | 使用场景 | 主要角色 | 典型链路 |

| **快捷操作**（销售汇总、人力汇总、菜单搭建、退款查询、工时录入等） | 将 **最高频后台动作** 暴露为 **捷径** | 不等同报表树检索，直接 **办事** | 店长、排班/人事、收银主管 | 首页 → 快捷项 → 对应功能页 |

| **按品项净销售额、明细分解** | **结构下钻** 入口 | 看 **哪些品项/时段** 贡献收入与人力效率 | 运营、厨师长（配合菜单） | 首页 → 卡片 → 明细页/报表 |
| **金融产品 / 顾客反馈 / Toast 在线点餐 / 影响分析 / Toast 推荐** | **交叉销售 + 粘性模块** 入口 | 引导使用 **金融、反馈、渠道、增长工具** | 老板、运营 | 首页 → 模块落地页 → 开通或配置 |

### 2.2.3 一级导航：报表

| 项目 | 内容 |
|------|------|
| **命名定义** | **Reports / 报表**：按 **业务主题** 组织的 **只读分析** 集合（非单据操作主入口）。 |
| **使用场景** | 对账、复盘、稽核、集团对比、合规留痕；**不**用于改菜单/改价（改在分组二对应模块）。 |
| **使用角色** | 店长、财务、总部运营、稽核；部分报表给 **厨师长/前厅** 看后厨与服务质量。 |
| **使用链路** | 报表 → **报表总览** → 选主题（销售/人力/支付…）→ 具体报表名 → 筛选门店、日期 → 导出（若允许）。 |

#### 报表下二级导航：「报表总览」主题簇

| 二级主题 | 命名意图 | 使用场景 | 主要角色 | 典型链路 |
|----------|----------|----------|----------|----------|
| **行业对标** | 与同行 **Benchmark** | 评估自身在细分市场的位置 | 老板、运营 | 报表 → 对标 → 筛选维度 → 解读差距 |

| **销售** | 收入与订单 **全链路** | 收入确认、渠道结构、大单追踪 | 店长、销售运营 | 销售汇总 → 明细 → 对异常订单下钻订单详情 |
| **人力** | 工时、合规、小费 | 劳动力成本与合规审计 | 人事、店长 | 人力汇总 → 工时/休息 → 与 Payroll 核对 |
| **菜单** | 品项 **结构与加料** | 菜单工程、沽清、损耗关联 | 运营、厨师长 | 品项结构 → 滞销/爆款 → 回菜单模块改价 |

| **现金与损耗管理** | 钱箱、作废、退款、免税等 | **现金安全** 与 **损耗稽核** | 店长、稽核 | 钱箱历史 → 与实点现金比对 |
| **账务** | 总账视角 | 与会计系统或税务衔接 | 财务 | 按日/按店账务 → 导出给总账 |
| **后厨运营** | 出餐时效、单据 | **厨房效率** 与 SLA | 厨师长 | 按小时分单 → 优化档口 |
| **营销** | 顾客与会员 **结果数据** | 活动 ROI、会员价值 | 市场、运营 | 顾客汇总 → 复盘活动 |
| **订位与等位** | 预订侧 **结果** | 上座率、预订质量 | 前厅经理 | 与「等位与订位」模块配置联动 |

## 2.3 分组二：按业务域划分的运营与配置中心

**分组定位：** 日常 **管人、管货、管钱、管客、管场** 的主体；**配置 + 部分报表（相关报表）** 并存。

**共性链路模式：** 进入模块 → **配置/事务** → 需要时 **相关报表** 验证效果 → 回到 **首页/报表** 看总览。

---

### 2.3.1 员工

| 层级 | 命名定义 | 使用场景 | 主要角色 | 典型链路 |
|------|----------|----------|----------|----------|
| **一级** | **Employees / 员工** | **人力全生命周期**（档案、排班、权限、考勤、小费规则） | 店长、HR、排班经理 | 员工 → 员工管理 → 编辑员工 → 岗位/权限 |
| **二级-员工管理** | 团队、员工档案、岗位、Sling 排班、加班、招聘物料 | **招人、排班、日常维护** | HR、店长 | 员工管理 → Sling → 发布班次 |
| **二级-权限/薪资/班次回顾/POS 报表配置/考勤** | 权限、Payroll、回顾模板、考勤修正 | **合规发薪、对账、审计** | HR、财务 | 考勤 → 工时录入管理 → 与 Payroll 审批衔接 |
| **二级-相关报表** | 人效、人力汇总、小费池、休息合规等 | **验证人力成本与合规** | 店长、HR | 员工 → 相关报表 → 与报表中心 **人力** 主题重叠（双入口） |
| **二级-团队聊天** | 店内沟通工具设置 | **协作与通知** | 店长、全员 | 较少与报表交叉 |

---

### 2.3.2 薪资（Payroll）

| 层级 | 命名定义 | 使用场景 | 主要角色 | 典型链路 |
|------|----------|----------|----------|----------|

## 2.7 三条典型端到端链路（跨导航）

1. **日结与对账**  
   首页看销售额 → **报表-支付/现金** 明细 → 发现差异 → **支付** 模块查单退款 → **Toast 账户** 看入账银行。

2. **上新与验证**  
   **菜单** 改价发布 → **前厅/外带** 侧可见 → **报表-销售/菜单** 看品项表现 → **营销** 配合同步活动。

3. **订位体验闭环**  
   **等位与订位** 配短信与政策 → 顾客到店 → **前厅 Toast 桌边** 开台 → **报表-订位与等位** 看转化；异常 **顾客反馈** 回流。
